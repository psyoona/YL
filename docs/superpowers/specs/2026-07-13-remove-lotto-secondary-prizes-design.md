# Lotto 2등·3등 노출 제거 설계

## 목표

`Laboratory/Lotto`의 모든 사용자 노출 경로에서 2등과 3등 관련 데이터와 기능을 제거한다. 동행복권 신규 파일에서 더 이상 제공되지 않는 값에 UI와 API가 의존하지 않도록 하면서, 다른 프로그램과 기존 데이터를 위해 데이터베이스의 `Reward2`, `Reward3` 컬럼은 유지한다.

## 범위

- 결과 목록 표에서 2등·3등 상금 열을 제거한다.
- CSV 내보내기에서 2등·3등 상금 헤더와 값을 제거한다.
- 당첨금 계산기에서 2등·3등 선택지와 계산 분기를 제거한다.
- 생성 번호의 과거 당첨 이력은 1등 일치만 판정하고 표시한다.
- Lotto API 응답 모델에서 `Reward2`, `Reward3`을 제거한다.
- 평균 당첨금 API는 1등 평균만 조회하고 반환한다.
- 2등·3등 전용 화면 스타일과 클라이언트 표시 매핑을 제거한다.

## 제외 범위

- 데이터베이스 스키마나 기존 행을 변경하지 않는다.
- LottoDrawer의 저장 로직을 변경하지 않는다.
- 1등, 4등, 5등 계산과 다른 번호 통계 기능은 변경하지 않는다.

## 서버 설계

`LottoInformation`에서 `Reward2`, `Reward3` 속성을 제거한다. `LottoService`의 모델 바인딩 쿼리는 `SELECT *` 대신 회차, 추첨일, 번호, 보너스, 1등 상금과 필요한 행 번호만 명시적으로 선택한다. 이렇게 하면 신규 행의 `NULL` 2등·3등 값이 비-nullable DTO에 바인딩되는 경로도 함께 제거된다. 단건 조회도 열 이름을 그대로 속성에 연결하는 목록 Binder를 사용해 `NumberBonus`와 같은 복합 속성명이 손상되지 않게 한다.

`CheckNumberCombination`은 여섯 번호가 모두 일치하는 1등만 결과로 만든다. `GetAverageRewards`는 `Reward1` 평균만 계산하고 `AVG_REWARD_1`만 반환한다. 기존 API 라우트는 유지하되 응답에서 2등·3등 필드는 제외한다.

## 클라이언트 설계

Razor 결과 표는 총 10열로 축소하고 빈 결과 행의 `colspan`도 10으로 맞춘다. CSV는 회차, 추첨일, 여섯 번호, 보너스, 1등 상금만 내보낸다.

당첨금 계산기의 매칭 선택지는 1등, 4등, 5등만 유지한다. 생성 번호 이력의 등수 아이콘 매핑과 관련 CSS에서도 2등·3등 항목을 제거한다.

## 호환성과 오류 처리

DB 컬럼은 그대로 남아 LottoDrawer와 과거 데이터에 영향을 주지 않는다. YL은 해당 컬럼을 읽거나 직렬화하지 않으므로 값이 `NULL`이어도 목록, 최근 당첨 번호, 단건 조회가 정상적으로 모델에 바인딩된다. 공개 API에서 `REWARD2`, `REWARD3`이 사라지는 것은 이번 요구사항에 포함된 의도적인 계약 변경이다.

## 검증

- 실제 API의 대문자 JSON 정책으로 직렬화했을 때 `REWARD2`, `REWARD3`이 없고 `REWARD1`은 유지되는 회귀 테스트를 먼저 실패시킨다.
- 단건 조회가 안전한 목록 Binder를 사용하도록 하는 회귀 테스트와 실제 API 호출을 검증한다.
- Lotto 화면, JavaScript, 서비스에 2등·3등 표시 토큰과 `Reward2`, `Reward3` 조회 의존성이 남지 않는 계약 테스트를 먼저 실패시킨다.
- 전체 xUnit 테스트와 Debug/Release 빌드를 실행한다.
- 로컬 페이지를 열어 결과 표, CSV 구성, 계산기 선택지, 생성 번호 이력에 2등·3등 정보가 없는지 확인한다.

## 작업 제약

- 현재 `master`에서 직접 수정한다.
- 워크트리, 스테이징, 커밋을 생성하지 않는다.
- 기존 `Properties/PublishProfiles/ReleaseProfile.pubxml.user` 변경을 보존한다.
